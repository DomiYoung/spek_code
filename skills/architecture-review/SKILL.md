# Architecture Review Skill

## 优先级层次
```
长期可维护性 > 可扩展性 > 性能 > 短期收益
```

## SOLID 原则检查

### S - 单一职责
- 每个类/函数只有一个变更原因
- 检查：模块是否做了太多事情？

### O - 开闭原则
- 对扩展开放，对修改封闭
- 检查：新功能是否需要修改核心代码？

### L - 里氏替换
- 子类可以替换父类
- 检查：继承层次是否合理？

### I - 接口隔离
- 客户端不应依赖不需要的接口
- 检查：接口是否过于臃肿？

### D - 依赖倒置
- 依赖抽象，不依赖具体实现
- 检查：高层模块是否依赖低层细节？

## 架构审查清单

### 代码层面
- [ ] 模块职责清晰
- [ ] 依赖关系合理
- [ ] 接口定义简洁
- [ ] 可测试性良好

### 系统层面
- [ ] 组件边界明确
- [ ] 通信机制合理
- [ ] 错误处理完善
- [ ] 可观测性支持

## 核心设计原则
- **DRY**: 消除重复
- **KISS**: 保持简单
- **YAGNI**: 不做过度设计
- **组合优于继承**
- **高内聚低耦合**
